PART 1 | NoSQL & MongoDB 기초

웹 서비스 아키텍처 변화와 NoSQL의 도입 배경

  • 비정형 데이터의 증가: 로그, 센서 수집값, 소셜 미디어 피드 등 형태가 고정되지 않은 대용량 데이터의 적재와 빠른 처리가 필요해졌습니다.
  • 유연한 데이터 구조 요구: 서비스 배포 주기가 빨라짐에 따라, 관계형 데이터베이스(RDBMS)의 스키마 변경(DDL 실행)에 수반되는 리스크와 비용을 완화하고자 했습니다.
  • 수평 확장성(Scale-out): RDBMS는 주로 수직 확장(Scale-up)에 의존하지만, NoSQL은 분산 처리를 염두에 두고 설계되어 저렴한 노드를 확장하는 수평 확장에 유리합니다.
MongoDB는 데이터를 문서(Document) 단위로 묶어 저장합니다. 따라서 관계 정의보다 '어플리케이션에서 한 번에 어떤 단위로 데이터를 읽을 것인가'를 우선적으로 고민해야 합니다.

PART 1 | NoSQL & MongoDB 기초

RDBMS와 MongoDB의 설계 철학 비교

RDBMS (MySQL, PostgreSQL 등)

  • Database ➔ Table ➔ Row / Column
  • 데이터 정규화(Normalization)를 통한 중복성 최소화
  • 외래키(Foreign Key)와 JOIN을 사용해 데이터를 관계 매핑
  • 전체 테이블에 대한 ACID 트랜잭션 보장 수준이 높음

MongoDB

  • Database ➔ Collection ➔ Document / Field
  • 데이터 비정규화(Denormalization)를 통한 읽기 성능 우선
  • 연관 데이터를 하나의 문서에 내장(Embedding)하는 것을 권장
  • 단일 문서 내의 원자성(Atomicity)을 보장하며, 필요시 다중 문서 트랜잭션 사용

PART 1 | NoSQL & MongoDB 기초

분산 시스템과 CAP 정리

  • 분산 네트워크 환경에서 다음 3가지 특성을 동시에 모두 만족하는 것은 불가능하다는 이론입니다.
  • C (Consistency, 일관성): 분산 노드 중 어느 곳에 쿼리하더라도 항상 가장 최근에 업데이트된 동일한 데이터를 조회할 수 있어야 합니다.
  • A (Availability, 가용성): 일부 노드에 장애가 발생하더라도 정상 작동 중인 노드는 항상 클라이언트의 요청에 에러 없이 응답해야 합니다.
  • P (Partition Tolerance, 분할 내성): 노드 간 네트워크 단절이 발생하더라도 시스템 전체가 정상적으로 동작해야 합니다.
  • MongoDB의 아키텍처 방향성 (CP): 네트워크 분할 상황에서 가용성보다 데이터의 일관성을 우선시합니다. (예: Primary 장애 시 Replica Set의 투표 기간 동안 쓰기 불가능)

네트워크 장애 발생 시 데이터베이스의 선택

CA (가용성+일관성): 단일 노드 RDBMS (분산 네트워크 고려 없음)

AP (가용성+분할내성): Cassandra, DynamoDB (최종 일관성 보장)

CP (일관성+분할내성): MongoDB, HBase (일관된 데이터 제공 우선)

BSON(Binary JSON)의 특징과 구조

  • MongoDB는 네트워크 전송 및 디스크 저장 시 JSON이 아닌 BSON 형식을 사용합니다.
  • JSON의 구조적 한계: 텍스트 형식이므로 파싱 비용이 높고, 날짜(Date)나 상세 숫자(Int32, Decimal128 등) 타입을 기본적으로 구별하지 못합니다.
  • BSON의 설계상 장점:
    • 바이너리 형태로 직렬화되어 인코딩/디코딩 성능이 우수합니다.
    • 필드 길이와 요소의 위치가 저장되므로, 쿼리 수행 시 필요한 필드만 건너뛰며 스캔할 수 있습니다.
    • 날짜와 정밀한 숫자 연산을 위한 추가 타입을 내장하고 있습니다.
// BSON 명세에 따른 데이터 표현 예시 { "_id": ObjectId("507f191e810c19729de860ea"), "title": "BSON 구조의 이해", "createdAt": ISODate("2026-05-27T10:00:00Z"), "views": NumberInt(1500), "price": NumberDecimal("19.99"), "is_active": true }

문서(Document) 모델의 중첩 구조 표현

  • 관계형 데이터베이스에서는 다중 값을 저장할 때 데이터의 무결성을 위해 별도 테이블을 생성하고 JOIN 처리를 요구하는 것이 보편적입니다.
  • MongoDB는 배열(Array)과 중첩 문서(Embedded Document)를 네이티브 데이터 타입으로 지원합니다.
  • 애플리케이션 계층의 객체 모델을 영속성 계층에 직접 투영할 수 있어, 임피던스 미스매치(Impedance Mismatch) 현상을 줄여줍니다.
설계 권장 조건:
자주 같이 조회되고 갱신되는 관련 데이터 그룹은 데이터의 응집도를 위해 하나의 문서 구조로 통합(Embedding)하는 것을 우선 고려합니다.

유연한 스키마와 데이터 검증(Schema Validation)

  • MongoDB는 고정 스키마를 강제하지 않으므로 초기 요구사항 변경에 대응하기 쉽습니다.

PART 2 | 데이터 모델링 원칙

임베딩(Embedding)과 참조(Referencing)의 선택 기준

임베딩 (Embedding)

  • 관련 데이터를 단일 문서 내에 하위 요소로 배치합니다.
  • 장점: 데이터의 참조 비용이 없어 읽기 속도가 빠릅니다.
  • 단점: 16MB 크기 제한에 걸릴 위험이 있으며, 비정규화로 인한 갱신 비용이 큽니다.

참조 (Referencing)

  • 독립된 컬렉션에 저장한 뒤 _id 식별자로 조회합니다.
  • 장점: 정규화되어 데이터 정합성 관리가 쉽고, 문서 확장 한계를 방지합니다.
  • 단점: 데이터 통합 시 조인 연산($lookup) 또는 N+1 조회 쿼리가 수반됩니다.

1:1 관계 설계 패턴

  • 두 엔티티가 라이프사이클을 공유하며 상호 일대일로 맵핑되는 도메인 모델입니다.
  • 권장 방안: 데이터의 보안 등급 차이나 특정 필드의 용량(예: 바이너리 첨부 파일)이 문제되지 않는다면 임베딩 처리를 우선합니다.
  • 한 차례의 디스크 I/O 작업만으로 필요한 리소스를 수집하기 때문에 비효율적인 조인 작업을 줄일 수 있습니다.
{ "_id": "user_001", "name": "이경수", "profile": { // 1:1 결합 형태의 임베딩 "age": 25, "address": "전라북도 전주시 백제대로", "github": "github.com/macslab" } }

PART 2 | 데이터 모델링 원칙

1:N 관계: Cardinality(농도)에 따른 설계 구분

  • 관계형 모델은 외래키 정규화로 일괄 처리하지만, 문서형 데이터베이스는 관계의 성장 크기에 따라 다른 접근이 요구됩니다.

  • One-to-Few (소규모 관계): 데이터가 많아야 수십 개 미만인 경우. 하나의 부모 문서 내부에 임베딩 배열로 축적합니다. (예: 사용자의 배송지 목록)
  • One-to-Many (중규모 관계): 데이터가 수백~수천 개로 유한한 경우. 독립된 컬렉션으로 구현하되 부모 문서에서 자식 문서의 식별자 배열을 유지할 수 있습니다. (예: 카테고리-상품 리스트)
  • One-to-Squillions (대규모 관계): 데이터가 끊임없이 증가하고 수만 건 이상 늘어날 수 있는 경우. 부모의 식별자 배열이 16MB 상한선을 초과할 우려가 있으므로, 반드시 자식 문서 쪽에서 부모의 _id를 참조하도록 모델링합니다. (예: 특정 서버 로그)

1:N 관계: 적은 수의 관계 (임베딩)

  • 사용자의 복수 연락처나 거주지 정보 등 추가되는 데이터 개수가 예측 가능하고 한정적인 관계입니다.
  • 배열의 증가폭이 제한적이므로 문서가 비대해져 성능 저하나 한계를 유발할 여지가 상대적으로 적습니다.
  • 조회 성능 극대화와 단일 쓰기 작업을 보장받기에 유용합니다.
{ "name": "Alice", "addresses": [ { "type": "home", "city": "Jeonju" }, { "type": "work", "city": "Seoul" } ] }

PART 2 | 데이터 모델링 원칙

1:N 관계: 대규모 관계 (부모 참조)

  • 특정 장비의 모니터링 로그나 결제 트랜잭션 기록처럼 수시로 추가되며 양이 매우 많은 도메인 관계입니다.
  • 부모 문서(Host)에 자식 데이터 배열을 포함하는 경우, 얼마 지나지 않아 BSON 문서 용량 제한인 16MB를 충족하지 못해 에러가 유발됩니다.
  • 따라서 각 로그 문서가 부모 장비 문서의 식별자를 저장하게 함으로써 무제한적 성장을 회피합니다.
// Host Collection { "_id": "server_1", "ip": "192.168.0.1" }
// Logs Collection (부모 ID 참조) { "host_id": "server_1", "msg": "CPU Overload", "time": ISODate("2026-05-27T10:00:00Z") }

PART 2 | 데이터 모델링 원칙

N:M 관계: 다대다 관계 설계

  • 학습자와 교과목처럼 상호 다대다 구조를 이루는 테이블 매핑 구조입니다.
  • 관계형 스키마와 달리 연결 관계를 중간에서 매개하는 매핑 테이블 없이, 도메인 문서에 식별자 배열 필드를 직접 보유하는 형태로 설계 가능합니다.
  • 양방향 참조(Two-way Referencing): 양 개체의 성장 규모가 16MB 한도를 넘지 않는 조건 하에, 도메인 문서에 서로의 식별자 배열을 유지하여 양측 검색 속도를 개선합니다.
// Student Document (수강 목록 참조) { "name": "Alice", "enrolled_courses": ["CS101", "DB202"] }
// Course Document (수강생 목록 참조) { "_id": "DB202", "students": ["Alice_ID", "Bob_ID"] }

임베딩 패턴 (Embedding Pattern)

  • 핵심 목적: 단일 도메인 조회 시 수반되는 디스크 참조 횟수를 줄이고 네트워크 비용을 낮추는 구조입니다.
  • 적용 케이스: 일대일 대응이거나, 일대다 관계에서 다(Many) 측의 데이터 개수와 용량이 물리적으로 적을 때 유효합니다.
  • 설계 주의점: 갱신 빈도가 과다한 공유 자원을 내장하면 데이터의 불일치 현상이 초과로 발생할 위험성이 있습니다.
// 배송 정보와 항목을 포함하는 단일 주문 문서 { "order_id": "ORD-1234", "total_price": 55000, "shipping_address": { "zip": "12345", "street": "Seoul" }, "items": [ {"prod_id": 1, "qty": 2}, {"prod_id": 5, "qty": 1} ] }

참조 패턴 (Reference Pattern)

  • 핵심 목적: 공통 데이터의 중복 적재를 피해 갱신 이상을 막고 문서의 비대화를 예방하는 패턴입니다.
  • 적용 케이스: 도메인 개체들이 각각 자주 조회되거나, 부모-자식의 포함 차수(Cardinality)가 커 16MB 제한 준수가 힘들 때 활용합니다.
  • 도입에 따른 조인 비용과 쿼리 다수 실행 비용이 수반되는 정적 설계의 정형적 패턴입니다.
// 출판사 정보 (독립 컬렉션) { "_id": "pub1", "name": "O'Reilly" }

버킷 패턴 (Bucket Pattern): 시계열 데이터 최적화

  • 주요 해결 이슈: IoT 기기, 주식 거래 틱 데이터 같은 연속적 데이터 적재 시, 매번 별도 문서를 삽입하면 인덱스 오버헤드와 저장 공간 메타데이터 중복이 매우 극심해집니다.
  • 설계 방식: 일정한 기준 범위(예: 1일 단위 혹은 수집 1,000건 단위)별로 문서를 하나 생성하고, 그 내부 배열에 측정값을 차례대로 push하여 적합한 버킷을 구축합니다.
  • 데이터베이스 레벨의 쓰기 횟수가 감소하고 탐색 범위를 그룹 단위로 필터링하여 인덱스 크기를 대폭 감축합니다.

계산 패턴 (Computed Pattern): 사전 집계 연산

  • 핵심 이슈: 대규모 트래픽에서 게시글의 누적 조회수, 영화 평점 평균 등을 조회할 때마다 매번 실시간 계산을 수행하면 심각한 병목 현상이 발생합니다.
  • 해결 방식: 쓰기 시점의 비용을 조금 높이더라도, 연계 결과 데이터를 미리 집계하여 문서의 지정 필드에 기록해 둡니다.
  • 원자적 증감 연산자($inc) 등을 동원하여 동시성 문제를 제어하며 통계 정보를 갱신합니다.

부분집합 패턴 (Subset Pattern): 최신 데이터 우선 노출

  • 주요 배경: 메모리(RAM)에 활성 상태로 적재되는 데이터셋 영역인 Working Set의 부피가 커지면 디스크 Swap이 빈번해져 응답 속도가 급락합니다.

다형성 패턴 (Polymorphic Pattern)

  • 주요 배경: 게시글 양식의 다형성이나 정기 결제 수단(신용카드, 간편결제, 휴대폰 결제)처럼 기본 정보는 공통되지만 하위 필드 구조가 다를 때 발생합니다.
  • 관계형 데이터베이스에서는 각각의 하위 테이블을 분할하여 관계를 매핑한 뒤 JOIN 구조를 쓰게 되므로 스케일이 비효율적입니다.

속성 패턴 (Attribute Pattern): 동적 필드 검색 인덱싱

  • 핵심 배경: 의류의 사이즈/색상, 노트북의 RAM 용량/CPU 기종 등 상품 카테고리마다 검색이나 필터링 기준이 되는 특성이 가변적일 때 유용합니다.
  • 개별 특성을 단독 필드로 설계하면 인덱스 개수 상한에 도달하거나 메모리 부하가 심각해집니다.
  • 해결 방식: 속성 명칭(Key)과 속성값(Value)을 하나의 쌍으로 묶어 배열 형태로 담고, {"specs.k": 1, "specs.v": 1} 복합 인덱스를 1개만 생성하여 일관성 있게 관리합니다.

{ "specs": [ { "k": "cpu", "v": "M3" }, { "k": "ram", "v": "128GB" } ] } // db.products.createIndex({"specs.k": 1, "specs.v": 1})

인덱스(Index)의 기본 원리와 최적화

  • COLLSCAN vs IXSCAN: 인덱스가 없으면 컬렉션 전체 문서를 물리적으로 모두 순차 탐색하므로 검색 속도가 선형으로 하락합니다. 인덱스는 검색 값을 B-Tree 인덱스 파일로 정렬하여 조회 경로를 줄여줍니다.
  • 조회-쓰기 상호 절충(Tradeoff): 인덱스는 데이터 읽기 성능을 크게 늘리는 대신, 쓰기/갱신 작업 시마다 인덱스 트리를 지속적으로 수정하므로 오버헤드가 누적됩니다.
  • 따라서 쿼리 발생 패턴과 정렬(Sort) 작업의 기준 필드를 검토하여 선별적으로 인덱스를 개설해야 합니다.
// Mongoose 스펙에서의 단일/복합 인덱스 부여 예제 userSchema.index({ username: 1 }); // 복합 인덱스 (나이 내림차순 정렬 후 이름 오름차순 매칭 시 유효) userSchema.index({ age: -1, username: 1 });

$group GROUP BY 지정 키 기준 그룹화. 개수(sum), 평균(avg) 등의 누적 연산자를 수반합니다.
$project SELECT 출력 필드 지정. 신규 가공 필드 생성, 불필요 필드 억제(0) 처리를 담당합니다.
$sort / $limit / $skip ORDER BY / LIMIT / OFFSET 메모리 정렬 및 노출 범위 필터링. 정렬 작업은 적합한 인덱스 지원이 필요합니다.
$lookup LEFT JOIN 이종 컬렉션 조인. 관계 구조를 복원할 때 조인 결과 배열을 문서에 맵핑합니다.

  • 역할: 관계형 데이터베이스의 Outer Join과 동등한 기법으로, 기준 문서와 대상 컬렉션 문서 간의 관계 매핑을 처리합니다.
  • 조인 결과물은 명시한 식별 키(as)에 맞춰 배열 필드 형태로 결합되므로 단일 객체 매핑 시 후처리가 연이어 필요합니다.

과도한 참조 조인(Deep Join) 방지

  • 안티패턴: NoSQL 환경에서 정규화 편향에 의해 User, Post, Comment, Tag, Like 컬렉션을 개별로 나누고 조인을 수행하는 구조입니다.
  • 단일 피드 노출 작업 시 마이크로초 단위의 지연이 연쇄 누적되므로 데이터 조인의 비용이 급증하고 쿼리의 관리 가독성이 떨어집니다.
  • 대안적 정돈: 게시글의 부가 정보(예: 태그 리스트, 단순 좋아요 총량)는 Post 문서 내부에 내장시켜 한 번의 스캔으로 리소스를 획득하게끔 설계합니다.

정규화 강박의 탈피

무조건적인 분할 설계(Normalization)에서 벗어나, 도메인의 공통 읽기/쓰기 단위를 기준으로 문서를 통합 관리하는 것이 MongoDB 모델링의 핵심 가치입니다.

동시성 이슈를 고려한 추천(좋아요) 처리

  • 경쟁 조건(Race Condition): 수십 명의 사용자가 거의 동시에 좋아요를 눌렀을 때, 단순 읽기 후 수치의 연산은 잘못된 최종 합산 수치를 만듭니다.
  • 원자성 확보: 어플리케이션 계층에서의 중복성 확인 조건부 연산을 피하고, 단일 쿼리 조건 단계에서 해당 유저의 추천 여부를 걸러 원자적인 처리를 구현합니다.
  • 조건절에 likes: { $ne: userId }를 추가해 중복 추천을 원천 차단하며 like_count 증감을 단일 트랜잭션으로 체이닝합니다.
// 1. 추천(좋아요) 처리 - 원자적 조건 업데이트 const likeResult = await Post.updateOne( { _id: postId, likes: { $ne: userId } }, // 유저가 좋아요 안 누른 경우만 매칭 { $addToSet: { likes: userId }, $inc: { like_count: 1 } } ); // modifiedCount === 0 인 경우 이미 누른 상태임 // 2. 추천 취소(좋아요 취소) 처리 const unlikeResult = await Post.updateOne( { _id: postId, likes: userId }, // 유저가 기존에 누른 상태인 경우만 매칭 { $pull: { likes: userId }, $inc: { like_count: -1 } } );

계층형 댓글(대댓글) 데이터 모델링

  • 인접 리스트 (Adjacency List): 개별 댓글 문서에 직속 상위 댓글 식별자인 parent_id를 필드로 둡니다. 데이터 조회가 직관적이나 무한 중첩 트리 조회 시 N+1 쿼리 오버헤드가 발생할 수 있습니다.
  • 조상 경로 배열 (Array of Ancestors): 상위 노드들의 식별자 전체를 ancestors: [ObjectId("A"), ObjectId("B")]와 같이 배열로 포함합니다. 단일 질의로 하위 댓글 트리를 통째로 찾아오거나 인덱스를 타는 데 적절합니다.
  • 구체화된 경로 (Materialized Paths): path: "0001.0003.0012"와 같이 조상들의 경로 문자열을 기술합니다. 정규식 쿼리를 사용하여 특정 깊이 탐색 작업을 효율화합니다.
  • 모델 선정 권장 기준: 깊이가 1단계(댓글-답글)에 그칠 때에는 인접 리스트가 타당하며, 임의 깊이를 지원하는 트리형 스크롤 구조에서는 조상 경로 배열을 채용하는 방안이 안정적입니다.
