현대 백엔드 트렌드와 NoSQL의 부상

  • 폭발적인 데이터 증가: 빅데이터 시대, 비정형 데이터(로그, 센서 데이터, 소셜 미디어 피드)의 급증.
  • 애자일(Agile) 개발: 서비스 요구사항이 수시로 변경되면서, 고정된 테이블 스키마(RDBMS)의 변경 비용이 너무 높아졌습니다.
  • 클라우드와 스케일 아웃(Scale-out): RDBMS는 수직 확장(Scale-up, 비싼 서버 구매)에 의존하지만, NoSQL은 저렴한 서버를 여러 대 이어붙이는 수평 확장(Scale-out)에 최적화되어 있습니다.
"데이터를 엄격히 관리할 것인가, 유연하고 빠르게 처리할 것인가?"
MongoDB는 유연성과 확장성에 초점을 맞춘 대표적인 Document DB입니다.

RDBMS vs MongoDB: 아키텍처 비교

RDBMS (MySQL, PostgreSQL)

  • Database ➔ Table ➔ Row / Column
  • 데이터 정규화(Normalization) 중시
  • JOIN을 통해 여러 테이블의 데이터를 엮어냅니다.
  • 강력한 ACID 트랜잭션 보장 (모든 테이블에 걸쳐).

MongoDB

  • Database ➔ Collection ➔ Document / Field
  • 데이터 비정규화(Denormalization) 중시
  • JOIN을 최소화하고 관련된 데이터를 한 문서에 임베딩합니다.
  • 단일 문서 내 원자성(Atomicity) 보장. (최근 다중 문서 트랜잭션도 지원)

CAP 정리 (CAP Theorem)

  • 분산 시스템은 3가지 특성 중 최대 2가지만 동시에 가질 수 있다는 원리.
  • C (Consistency): 일관성. 모든 노드가 항상 동일한 데이터를 반환.
  • A (Availability): 가용성. 장애가 나지 않은 노드는 항상 응답.
  • P (Partition Tolerance): 분할 내성. 네트워크 단절에도 시스템 작동.
  • MongoDB의 포지션 (CP): 파티션 내성이 있을 때, 응답을 못 하더라도 일관성을 지키는 편을 택합니다.

분산 데이터베이스 선택 기준

CA: Oracle, MySQL (네트워크 분할 허용 안함)

AP: Cassandra, DynamoDB (결과적 일관성, 응답 우선)

CP: MongoDB, HBase (일관성 최우선, Replica Set 구조)

JSON vs BSON (Binary JSON)

  • MongoDB는 문서를 BSON 형식으로 디스크에 저장합니다.
  • JSON의 한계: 텍스트 기반이라 파싱이 느리고, 데이터 타입이 적음(정수와 실수를 구분 못함, 날짜 타입 없음).
  • BSON의 장점:
    • 이진(Binary) 포맷으로 인코딩/디코딩 속도가 빠릅니다.
    • Date, Raw Binary, ObjectId 등 풍부한 데이터 타입을 지원합니다.
    • 문서 내부를 스캔할 때 전체를 파싱할 필요 없이 필요한 부분만 빠르게 찾습니다.
// BSON의 풍부한 데이터 타입 예시 { "_id": ObjectId("507f191e810c19729de860ea"), "title": "BSON 이해하기", "createdAt": ISODate("2026-05-27T10:00:00Z"), "views": NumberInt(1500), "price": NumberDecimal("19.99"), "is_active": true }

Document 모델의 가장 큰 특징: 배열과 객체

  • RDBMS에서 여러 개의 값(예: 사용자의 전화번호 2개)을 저장하려면 별도의 Phone 테이블을 만들고 JOIN해야 합니다.
  • MongoDB는 배열(Array)과 중첩 객체(Nested Object)를 기본적으로 지원하므로, 데이터를 있는 그대로 직관적으로 저장합니다.
  • 이는 개발자가 객체지향 언어(Java, Node.js)에서 클래스나 객체를 다루는 방식과 100% 일치합니다. (Impedance Mismatch 감소)

스키마리스(Schema-less) vs 스키마 검증(Schema Validation)

  • MongoDB 컬렉션은 기본적으로 어떤 형태의 문서든 받아들이는 무형(Schema-less)입니다.

Embedding vs Referencing (중첩 vs 참조)

Embedding (중첩)

  • 관련 데이터를 문서 내부의 배열/객체로 저장
  • 장점: 단 한 번의 DB 조회로 데이터를 모두 읽어옴. 디스크 I/O 최적화.
  • 단점: 16MB 문서 한계 도달 가능성. 중복 데이터 변경 시 여러 문서를 업데이트해야 함.

Referencing (참조)

  • 관련 데이터를 다른 컬렉션에 두고 `ObjectId`로 연결
  • 장점: 데이터 정규화. 무한히 커지는 배열 문제 방지. 데이터 변경 시 한 곳만 수정.
  • 단점: 데이터를 읽어올 때 `$lookup` (조인)이 필요하여 추가 조회 쿼리 발생.

1:1 관계 (One-to-One) 설계

  • 사용자 프로필과 상세 주소 정보처럼 1:1로 매칭되는 데이터.
  • 권장: 무조건 Embedding을 사용합니다.
  • 별도의 컬렉션으로 빼서 관리할 이유가 전혀 없습니다. MongoDB의 철학은 1번 읽을 때 필요한걸 다 가져오는 것입니다.
{ "_id": "user_001", "name": "이경수", "profile": { // 1:1 관계의 완벽한 임베딩 "age": 25, "address": "전라북도 전주시 백제대로", "github": "github.com/macslab" } }

PART 2 | Core Data Modeling Principles

1:N 관계 설계: 규모에 따른 분류

  • RDBMS는 1:N 관계를 항상 외래키(Foreign Key)로 처리하지만, MongoDB는 N의 크기(Cardinality)에 따라 전략을 3가지로 나눕니다.

  • One-to-Few (N이 작음, 수십 개 이하): 완벽히 Embedding. (예: 블로그 글에 달린 태그 목록)
  • One-to-Many (N이 중간, 수천 개): 자식(Many) 문서를 분리하고 부모(One) 문서에서 자식들의 `ObjectId` 배열을 가지거나 그 반대로 함 (참조). (예: 카테고리와 상품)
  • One-to-Squillions (N이 엄청나게 큼, 수백만 개): 부모 쪽에 배열을 두면 16MB 한계에 터짐. 무조건 자식 쪽에서 부모 ID를 가리키게 설계 (Parent Referencing). (예: 서버 로그)

1:N - One-to-Few (Embedding)

  • 사용자의 연락처나 주소가 2~5개 정도 있는 경우.
  • 배열의 크기가 제한적이므로 16MB 제한을 넘을 위험이 전혀 없습니다.
{ "name": "Alice", "addresses": [ { "type": "home", "city": "Jeonju" }, { "type": "work", "city": "Seoul" } ] }

1:N - One-to-Squillions (Parent Referencing)

  • 특정 장비(Host)가 쏟아내는 수백만 개의 로그 데이터.
  • Host 문서 안에 `logs: []` 배열을 만들면 하루 만에 16MB를 초과하여 터집니다.
  • 따라서 RDBMS처럼 로그 문서가 Host의 ID를 가지도록 설계해야 합니다.
// Host Collection { "_id": "server_1", "ip": "192.168.0.1" }
// Logs Collection { "host_id": "server_1", // 부모를 참조 "msg": "CPU Overload", "time": ISODate("2026-05-27T10:00:00Z") }

N:M 관계 (Two-way Referencing)

  • 학생(Student)과 강의(Course)의 다대다 관계.
  • RDBMS는 교차 테이블(Mapping Table)을 강제로 만들어야 하지만, MongoDB는 배열 덕분에 매핑 테이블이 필요 없습니다!
  • 학생 문서에는 수강 중인 `course_ids[]` 배열을 넣고, 강의 문서에는 `student_ids[]` 배열을 양방향으로 유지합니다.
// Student Document { "name": "Alice", "enrolled_courses": ["CS101", "DB202"] }
// Course Document { "_id": "DB202", "students": ["Alice_ID", "Bob_ID"] }

PART 3 | Advanced Schema Design Patterns

1. Embedding Pattern (복습)

  • 목적: 쿼리 횟수를 줄이고 읽기 속도를 극대화.
  • 적용 시기: 1:1 관계이거나 1:N 관계에서 N이 크지 않을 때.
  • 특징: 가장 기본적이며 NoSQL의 장점을 잘 살림. 하지만 N이 지속적으로 성장하는 데이터(예: 댓글)에는 위험함.
// Order Document Example { "order_id": "ORD-1234", "total_price": 55000, "shipping_address": { "zip": "12345", "street": "Seoul" }, "items": [ {"prod_id": 1, "qty": 2}, {"prod_id": 5, "qty": 1} ] }

2. Reference Pattern (복습)

  • 목적: 데이터 중복을 피하고 16MB 용량 초과 방지.
  • 적용 시기: 양쪽 모두 독립적으로 조회되거나 N이 엄청나게 클 때.
  • RDBMS의 정규화 방식과 가장 유사. Mongoose populate()나 MongoDB $lookup으로 결합.
// Publisher { "_id": "pub1", "name": "O'Reilly" }
// Book { "title": "MongoDB The Definitive Guide", "publisher_id": "pub1" // Reference }

3. Bucket Pattern (버킷 패턴)

  • 목적: 시계열 데이터(Time-series) 처리 시 인덱스 크기와 스토리지 낭비를 최소화.
  • 센서에서 1초마다 데이터가 들어올 때, 매초마다 새로운 문서를 만들면 메타데이터 중복과 문서 수가 폭발적으로 증가합니다.
  • 해결책: 데이터를 하루 또는 한 시간 단위로 배열 안에 묶어서(Bucket) 저장.
{ "sensor": "Temp_01", "start_date": ISODate("2026-05-27T00:00:00Z"), "end_date": ISODate("2026-05-27T23:59:59Z"), "measurements": [ {"t": 10, "val": 24.1}, {"t": 11, "val": 24.5} // ... 오늘 발생한 데이터 모음 ] }

4. Computed Pattern (계산 패턴)

  • 목적: CPU를 많이 소모하는 집계 연산을 피하기 위해.
  • 대시보드(통계 화면)처럼 조회가 매우 빈번하지만, 즉시 계산하기 무거운 경우 미리 결과를 계산해서 필드에 저장해 둡니다.
  • 새로운 데이터가 들어올 때 `$inc` (증가) 연산자로 필드 값을 갱신.

{ "title": "Inception", "total_reviews": 15000, // Computed "total_score": 72000, // Computed "avg_rating": 4.8 // Computed }

5. Subset Pattern (부분집합 패턴)

  • 목적: 워킹 셋(Working Set, 메모리에 올라가야 할 자주 쓰이는 데이터) 크기를 줄이기 위함.
  • 상품 상세 페이지를 열 때, 전체 10만 개의 리뷰를 다 볼 필요는 없습니다. 최신 리뷰 10개만 화면에 노출합니다.
  • 따라서 가장 많이 조회되는 일부(Subset) 데이터만 부모 문서에 임베딩하고, 나머지는 분리합니다.
// Product Doc (가장 자주 조회) { "name": "iPhone 17", "recent_reviews": [ { "text": "Great!", "rating": 5 } // 최신 10개만 유지 (Subset) ] }
➔
// Reviews Collection (더 보기 클릭 시) { "prod_id": "iPhone 17", "text": "Not bad", "rating": 3 }

6. Polymorphic Pattern (다형성 패턴)

  • 목적: 구조가 조금씩 다른 객체들을 하나의 컬렉션에 보관하여 검색을 단순화.
  • 페이스북 피드(Feed)에는 텍스트 게시물, 사진첩, 동영상, 광고 등 다양한 형태의 문서를 시간순으로 정렬해야 합니다. RDBMS라면 JOIN이 복잡해집니다.
  • MongoDB는 type 필드만 하나 두고, 서로 다른 스키마를 가진 문서를 한 컬렉션에 넣으면 됩니다!
// 단일 피드 컬렉션 내의 다형성 [ { "type": "text", "content": "Hello world" }, { "type": "image", "image_url": "http://img.com/1.jpg", "filter": "sepia" }, { "type": "ad", "sponsor": "Nike", "link": "..." } ]

7. Attribute Pattern (속성 패턴)

  • 목적: 필드 이름이 동적으로 변하는 경우 인덱스 생성을 위해.
  • 이커머스 상품 필터: 어떤 건 '용량', 어떤 건 '색상', 어떤 건 '주사율'을 필터링해야 합니다.
  • 동적 필드({ "color": "red" })를 만들면 인덱스를 걸 수 없습니다. Key-Value 형태의 배열로 묶어야 합니다.

Bad (인덱스 불가)

{ "cpu": "M3", "ram": "128GB" }

Good (Attribute Pattern)

{ "specs": [ { "k": "cpu", "v": "M3" }, { "k": "ram", "v": "128GB" } ] } // db.products.createIndex({"specs.k": 1, "specs.v": 1})

Aggregation Pipeline이란?

  • 다수의 문서를 처리하고 연산하여 복잡한 결과를 반환하는 프레임워크입니다.
  • UNIX의 파이프라인(|) 개념과 동일하게 작동합니다. 이전 단계(Stage)의 결과물이 다음 단계의 입력으로 전달됩니다.
  • RDBMS의 GROUP BY, JOIN, HAVING 등에 해당하는 고급 연산을 수행할 수 있습니다.
전체 주문(Orders)
➔
$match
(완료된 주문만 필터링)
➔
$group
(상품별 총합 계산)
➔
$sort
(많이 팔린 순 정렬)

주요 스테이지 (Stages) 요약

스테이지 SQL 매핑 설명
$match WHERE 특정 조건에 맞는 문서만 필터링합니다. (가급적 파이프라인 맨 앞에 두어 데이터 양을 줄여야 함)
$group GROUP BY 특정 키를 기준으로 문서를 그룹화하고 개수, 합계, 평균을 계산합니다.

$sort / $limit / $skip ORDER BY / LIMIT / OFFSET 정렬하고, 결과 개수를 제한하며, 건너뜁니다 (페이지네이션).
$lookup LEFT OUTER JOIN 다른 컬렉션의 데이터를 조인하여 합칩니다.

$lookup을 활용한 조인 (Join)

  • MongoDB는 비정규화(임베딩)를 권장하지만, 실무에서는 부득이하게 조인이 필요한 경우가 발생합니다.

$unwind: 배열 풀기

  • $lookup 결과는 항상 배열(`[]`)로 반환됩니다.
  • 만약 배열 내의 객체들을 각각 독립적인 문서로 분리해서 다루고 싶다면 $unwind를 사용합니다.
  • 예시: 사용자가 작성한 글들의 평균 길이를 구하고 싶을 때 배열 상태로는 불가능하므로 unwind한 뒤 `$group`을 적용합니다.
Before $unwind
{ user: "A", items: ["apple", "banana"] }
➔
After $unwind
{ user: "A", items: "apple" }
{ user: "A", items: "banana" }

읽기 중심 최적화 (비정규화 사례)

  • MongoDB는 빠른 읽기를 위해 비정규화(Denormalization) 및 데이터 중복을 적극적으로 활용합니다.

주의점: 사용자 닉네임이 변경되면 해당 사용자가 작성한 모든 게시글을 찾아 업데이트해야 하므로 쓰기 로직이 다소 복잡해집니다. 하지만 전체 트래픽 중 읽기가 압도적으로 많기 때문에 유리한 선택입니다.

딥 조인(Deep Join) Anti-Pattern 피하기

  • User → Post → Comment → Like 까지 모든 것을 독립적인 컬렉션으로 분리하면?
  • 읽기 요청 한 번에 $lookup이 연쇄적으로 발생하여 조회 속도와 쿼리 가독성이 급격히 붕괴됩니다.
  • 대안: 적절한 임베딩을 혼합해야 합니다. 태그(Tag)나 좋아요(Like) 목록은 Post 내부에 임베딩하는 것이 좋습니다.

대규모 댓글 설계 (Pagination 고려)

  • 만약 댓글이 게시물당 수만 개 달릴 수 있다면? 임베딩(comments: []) 시 16MB 한계를 터뜨리게 됩니다.

동시성을 고려한 좋아요(Like) 시스템

  • 사용자가 "좋아요"를 누를 때, 1명당 1번만 가능해야 합니다.
  • 동시에 수백 명이 접근할 경우 어플리케이션 단에서 검사하면 Race Condition (동시성 문제)이 발생합니다.
  • 해결책: 배열을 Set 자료구조처럼 취급하는 $addToSet 연산자를 사용합니다!
// 좋아요를 클릭했을 때 원자적(Atomic) 업데이트 const result = await Post.updateOne( { _id: postId, likes: { $ne: userId } }, { $addToSet: { likes: userId }, $inc: { like_count: 1 } } ); // result.modifiedCount === 0 이면 이미 좋아요를 누른 사용자

대댓글(Nested Comment) 설계 전략

  • 방법 1: Parent Referencing - 각 댓글이 부모 댓글의 ID를 가짐. (조회 시 트리를 직접 만들어야 하므로 복잡함)
  • 방법 2: Array of Ancestors (조상 배열) - ancestors: ["A", "B"] 배열로 모든 조상 저장.
  • 방법 3: Materialized Paths (구체화된 경로) - path: "001.002.003" 처럼 문자열 경로 저장 (정규식 검색 용이).
  • 추천: 댓글 깊이가 1단계(댓글 - 대댓글)로 얕다면 Parent Referencing이 낫고, 무한 깊이라면 조상 배열이나 구체화된 경로를 권장합니다.
