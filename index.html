PART 1 | Fundamentals

왜 MongoDB인가? (Philosophy)

  • Developer Productivity: 객체 지향 언어의 데이터 구조(JSON/BSON)와 매핑이 쉬움.
  • Scale-out Architecture: 서비스 중단 없이 수평적 확장(Horizontal Scaling) 가능.
  • Rich Query Language: 단순 Key-Value를 넘어 복잡한 분석 및 검색 지원.
  • Dynamic Schema: 비즈니스 요구사항 변화에 유연하게 대응 가능.
"데이터를 애플리케이션이 사용하는 방식 그대로 저장한다"는 것이 핵심입니다.

BSON (Binary JSON) 이해하기

  • JSON은 텍스트 기반으로 가독성이 좋으나 검색/연산 성능에 한계.
  • BSON은 MongoDB의 내부 저장 포맷으로 바이너리 구조.
  • 추가 데이터 타입 지원: ObjectId, Date, BinData(이미지 등), Decimal128 등.
  • 효율적인 인덱싱과 트래버스(Traverse)를 위해 설계됨.
// BSON Example { "_id": ObjectId("60d5f..."), "price": NumberDecimal("19.99"), "active": true, "created_at": ISODate("2026-05-27T...") }

Document Structure (Embedding)

중첩(Embedding)의 장점

  • One-Stop Read: 한 번의 I/O로 모든 관련 데이터를 조회.
  • Locality: 연관된 데이터가 물리적으로 가까운 곳에 위치.
  • Atomic Write: 하나의 문서는 항상 원자적 업데이트 보장.
{ "user": "kyungsoo", "addresses": [ { "city": "Jeonju", "zip": "54896" }, { "city": "Seoul", "zip": "06123" } ] }

Document Structure (Referencing)

참조(Referencing)의 장점

  • Data Consistency: 중복 데이터가 없어 수정 시 유리.
  • Flexibility: 다수의 관계가 복잡할 때 유연함.
  • Avoid 16MB Limit: 문서 하나가 너무 커지는 것을 방지.
// Orders Collection { "order_no": 101, "user_id": ObjectId("60d5f..."), "product_ids": [ ObjectId("77b21...") ] }

데이터 설계 원칙: Read vs Write

Read-Optimized (일반적)

조회 성능을 위해 비정규화를 허용하고 데이터를 중첩함.

GridFS: 대용량 파일 저장

  • MongoDB 문서 크기 제한은 16MB.
  • 이를 초과하는 파일(고화질 영상, 큰 PDF 등)을 저장하기 위한 명세.
  • 파일을 작은 청크(Chunks, 기본 255KB)로 나누어 별도 컬렉션에 저장.
  • 장점: 파일 시스템과 DB를 일원화하여 관리 가능, 샤딩 지원.

Insert & Save Operations

// Single Insert
db.users.insertOne({
  name: "Lee",
  age: 30

// Bulk Insert (추천)
db.users.insertMany([
  { name: "Kim", age: 25 },

Update Operators (Atomic)

  • $set: 특정 필드만 수정하거나 추가.
  • $inc: 숫자 값 증가/감소 (조회수, 추천수 등).
  • $push / $pull: 배열 요소 추가/삭제.
  • $unset: 필드 자체를 삭제.
db.products.updateOne(
  { _id: 101 },
  { $inc: { stock: -1 }, $set: { updated_at: new Date() } }
);

PART 2 | CRUD & Indexing

Indexing: 성능의 핵심

  • 인덱스가 없으면 Full Collection Scan (COLLSCAN) 발생.
  • Single Field Index: 특정 필드 하나에 생성.
  • Compound Index: 여러 필드 조합. (순서가 매우 중요: ESR 법칙)
  • Multikey Index: 배열 필드 내 요소를 인덱싱.
db.users.createIndex({ "email": 1 }, { unique: true });
16 / 50
PART 2 | CRUD & Indexing

Compound Index & ESR Rule

Equality ➔ Sort ➔ Range (ESR)

1. Equality: 정확히 일치하는 조건 (e.g., status: "active")

2. Sort: 정렬이 필요한 필드

3. Range: 범위 조건 (e.g., price > 100)

이 순서를 지키지 않으면 인덱스 효율이 급격히 떨어집니다.

PART 2 | CRUD & Indexing

TTL & Partial Indexes

  • TTL Index: 일정 시간 후 문서를 자동 삭제 (로그, 세션 등). db.logs.createIndex({ "createdAt": 1 }, { expireAfterSeconds: 3600 })
  • Partial Index: 조건에 맞는 문서만 인덱싱하여 인덱스 크기 절약. db.users.createIndex({ email: 1 }, { partialFilterExpression: { email: { $exists: true } } })

PART 3 | Design Patterns

Pattern 1: Attribute Pattern

  • 필드 이름이 불규칙하거나 검색 조건이 가변적일 때 사용.
  • 필드들을 key-value 쌍의 배열로 변환.
  • 장점: 인덱스 하나로 수많은 속성 검색 가능.
{ "attrs": [ { "k": "color", "v": "red" }, { "k": "size", "v": 10 } ] }
20 / 50
PART 3 | Design Patterns

Pattern 2: Extended Reference Pattern

  • 다른 컬렉션의 데이터를 참조하되, 자주 쓰이는 정보는 미리 복사해 둠.
  • 조인 횟수를 줄여 읽기 성능 극대화.
  • 원본 데이터 변경 시 업데이트 전파(Fan-out)가 필요함.
{
  "order_id": 1,
  "user_info": { // 참조 + 요약 정보 포함
     "id": ObjectId("..."),
     "name": "Lee"
  }
}
21 / 50
PART 3 | Design Patterns

Pattern 3: Subset Pattern

  • 1:N 관계에서 N이 너무 클 때(Working Set 문제), 최근 데이터 일부만 중첩하고 나머지는 참조로 분리.
  • 예: 게시물의 최근 댓글 10개만 Post에 보관, 나머지는 별도 조회.
  • 메모리 히트율(Hit Rate)을 높이는 전략.
22 / 50
PART 3 | Design Patterns

Pattern 4: Bucket Pattern (Deep Dive)

  • 시계열 데이터 최적화.
  • 단일 문서당 인덱스 오버헤드를 줄임.
  • 쿼리 결과의 크기를 적절히 유지.
{ "dev_id": 101, "bucket_start": "12:00", "samples": [ {"t": 0, "v": 20}, {"t": 1, "v": 21} ... ], "avg": 20.5 }

Pattern 5: Schema Versioning Pattern

  • 다운타임 없는 데이터 마이그레이션.
  • 문서에 schema_version 필드를 추가.
  • 애플리케이션이 구버전과 신버전 문서를 모두 핸들링할 수 있게 함.
  • 배치 작업으로 점진적 마이그레이션 수행 가능.

PART 3 | Design Patterns

Pattern 6: Computed Pattern (Best Case)

  • 읽기가 압도적인 데이터(통계, 카운터)에 적용.
  • 성능 부하가 큰 집계(Aggregation)를 쓰기 시점에 분산 처리.
  • 사용자 경험(UX) 개선의 핵심.
db.users.updateOne(
  { _id: 123 },
  { $inc: { total_points: 50, visit_count: 1 } }
);

Pattern 7: Outlier Pattern

  • 대부분의 데이터는 규칙을 따르지만 일부 예외(슈퍼스타, 헤비 유저)가 있을 때.
  • 예외 케이스를 위한 별도의 플래그나 오버플로우 문서를 생성.
  • 일반적인 데이터의 성능 저하를 방지.

Anti-Pattern: 잦은 $lookup

  • MongoDB는 관계형 DB가 아닙니다.
  • 성능의 핵심은 "Data Locality"입니다.
  • 3개 이상의 컬렉션을 $lookup으로 잇고 있다면 모델링 실패의 징조입니다.
정규화의 유혹을 뿌리치고 비정규화를 두려워하지 마세요.

PART 4 | Advanced Modeling

Multi-Document Transactions

  • MongoDB 4.0 이상부터 ACID 트랜잭션 지원.
  • 여러 문서나 컬렉션에 걸친 작업을 하나의 논리적 단위로 묶음.
  • 주의: 성능 오버헤드가 있으므로 반드시 필요한 경우만 사용.
const session = db.getMongo().startSession();
session.startTransaction();
try {
  db.accounts.updateOne({ id: 1 }, { $inc: { bal: -100 } }, { session });
  db.accounts.updateOne({ id: 2 }, { $inc: { bal: 100 } }, { session });
  session.commitTransaction();
} catch (e) {
  session.abortTransaction();
}

Change Streams: 실전 활용

  • 데이터의 삽입/수정/삭제를 실시간으로 구독.

PART 4 | Advanced Modeling

Working Set & RAM 관리

  • MongoDB 성능의 핵심은 "Working Set이 RAM에 들어오는가?"입니다.
  • 인덱스와 자주 쓰이는 데이터는 항상 메모리에 로드되어야 함.
  • Page Fault가 잦아지면 Disk I/O 병목으로 성능 급락.
"인덱스 크기(Index Size)는 항상 RAM 가용 용량보다 작아야 합니다."

Sharding: 데이터 분산 저장

  • 데이터를 여러 서버(Shard)에 나누어 저장.

Capped Collections

  • 크기가 고정된 특수 컬렉션.

  • 인덱스가 없어도 삽입 순서로 매우 빠르게 조회 가능.
