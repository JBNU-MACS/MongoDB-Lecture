PHASE 1 | Internals

Storage Engine: WiredTiger

  • Document-Level Concurrency: 여러 클라이언트가 동시에 다른 문서를 수정 가능 (No Lock).
  • Compression: Snappy/Zlib 알고리즘으로 스토리지 비용 30-80% 절감.
  • Checkpointing: 60초마다 데이터를 디스크에 플러시하여 데이터 영속성 보장.
  • Snapshot: 읽기 작업 시 일관된 데이터 뷰 제공.
~80%

PHASE 1 | Internals

Write Concern (w) & Journaling (j)

  • w: 1 - Primary 노드에만 기록되면 성공 (빠름).
  • w: majority - 과반수 노드에 기록 보장 (안전).
  • j: true - 메모리뿐만 아니라 Journal 로그 파일까지 기록 (크래시 대응).
db.users.insertOne( { name: "Alice" }, { writeConcern: { w: "majority", j: true, wtimeout: 5000 } } )

PHASE 1 | Internals

Read Preference & Consistency

Primary (Default)

항상 최신 데이터를 읽음. 쓰기가 몰리면 읽기 성능 저하 가능.

SecondaryPreferred

읽기 부하 분산(Read Scaling). 단, 복제 지연(Replication Lag)으로 인한 과거 데이터 가능성 존재.

"분산 시스템에서 일관성(Consistency)과 가용성(Availability)의 트레이드오프를 결정하는 핵심 설정입니다."

PHASE 1 | Internals

BSON: Type System Deep Dive

  • ObjectId: 12-byte (Timestamp + MachineID + PID + Counter).
  • Date: 64-bit 정수 (Unix Epoch ms).
  • Decimal128: 정확한 소수점 계산 (금융 서비스 필수).
  • Null vs Missing: 필드가 null인 것과 아예 없는 것은 다르게 취급됨.
{ "_id": ObjectId("60d5f..."), "balance": NumberDecimal("100.50"), "status": null, // "deleted_at" 필드 미존재 (Missing) }

PHASE 1 | Internals

The Power of Locality

Normalization (RDBMS)

Data scattered across tables. High disk seek time.

Denormalization (MongoDB)

Data grouped in a single block. Sequential I/O. Fast read.

"CPU는 기다려주지 않습니다. 데이터가 가까이 있어야 합니다."

PHASE 1 | Internals

Replica Set OpLog (Operations Log)

  • Primary의 모든 데이터 변경 사항이 기록되는 Capped Collection.
  • Secondary는 이 OpLog를 비동기적으로 복제하여 동기화.
  • Change Streams의 기술적 기반.
  • 복제 지연이 발생하면 OpLog가 덮어씌워져 'Stale' 상태가 될 수 있음.

PHASE 2 | CRUD

BulkWrite: High-Throughput Ingestion

  • 수천 개의 쓰기 작업을 하나의 네트워크 패킷으로 묶어 처리.
  • 네트워크 왕복(RTT) 비용을 획기적으로 절감.
  • Insert, Update, Delete를 한 번에 실행 가능.
db.collection.bulkWrite([
  { insertOne: { document: { _id: 1 } } },
  { updateOne: { filter: { _id: 2 }, update: { $set: { val: 10 } } } }
]);

PHASE 2 | Indexing

Partial & Sparse Indexes

Partial Index

특정 조건(예: rating > 4) 문서만 인덱싱. 저장공간 및 성능 최적화.

Sparse Index

필드가 존재하는 문서만 인덱싱. NULL 필드 제외 시 유용.

PHASE 2 | Indexing

Hashed & Text Indexes

  • Hashed Index: Shard Key의 고른 분산을 위해 필드 값을 해싱.
  • Text Index: 단순 키워드 검색 지원 (Atlas Search 도입 전 대안). db.posts.createIndex({ content: "text" })

PHASE 2 | Indexing

Multikey Index (Array Indexing)

  • 배열의 각 요소를 개별 인덱스 항목으로 저장.
  • 배열 내 필터링 속도를 극대화.
  • 주의: 배열 내 문서가 많으면 인덱스 크기가 폭발적으로 증가.
// tags: ["A", "B"] 인 경우 // 인덱스에 "A"와 "B"가 각각 기록됨 db.posts.createIndex({ "tags": 1 });

Index Building: Foreground vs Background

PHASE 3 | Design Patterns

Pattern 1: Attribute Pattern

언제 쓰는가?

  • 필드 이름이 너무 많거나 불규칙할 때.
  • 수백 개의 속성을 인덱싱해야 할 때.

➔ {"key": "val"} 배열로 구조화.

{ "specs": [ { "k": "battery", "v": "5000mAh" }, { "k": "screen", "v": "OLED" } ] } // Index: specs.k_1_specs.v_1
20 / 50
PHASE 3 | Design Patterns

Pattern 2: Extended Reference Pattern

  • Concept: "Join is expensive, so copy what you need."
  • 자주 조회되는 필드(예: 작성자 이름)를 문서 내에 복사.
  • Pros: No Lookups. Cons: Stale data risk.
{
  "post": "Hello MongoDB",
  "author": { "_id": 1, "name": "Lee" } // 복사된 정보
}
21 / 50
PHASE 3 | Design Patterns

Pattern 3: Subset Pattern

  • Concept: "Most people only care about the latest."
  • 게시글의 **최근 댓글 5개**만 Post에 포함, 나머지는 별도 컬렉션.
  • Working Set 크기를 줄여 메모리 활용 최적화.
22 / 50
PHASE 3 | Design Patterns

Pattern 4: Bucket Pattern

언제 쓰는가?

  • 초 단위 로그 데이터.
  • 인덱스 크기 폭발 방지.

➔ 데이터가 들어올 때 기존 문서를 찾아 $push.

{ "sensor_id": 1, "hour": "2026-05-27:14", "readings": [25.1, 25.2, 25.4...] }

PHASE 3 | Design Patterns

Pattern 5: Computed Pattern

  • Concept: "Do the math on write, not on read."
  • 조회수, 좋아요 수, 평균 별점 등을 필드로 미리 유지.
  • 집계 연산 부하를 0으로 만듦.
db.posts.updateOne(
  { _id: 123 },
  { $inc: { view_count: 1 } }
);

PHASE 3 | Design Patterns

Pattern 6: Schema Versioning Pattern

  • 비즈니스 로직 변화로 스키마가 변경될 때.
  • 기존 데이터를 모두 한 번에 바꿀 수 없을 때(Downtime 방지).
  • schema_version: 2 필드를 보고 애플리케이션에서 분기 처리.
25 / 50
PHASE 3 | Design Patterns

Pattern 7: Outlier Pattern

  • 대부분의 문서는 작지만 특정 문서(예: 인기글)가 16MB를 위협할 때.
  • 해당 문서만 has_overflow: true 플래그를 달고 추가 컬렉션 활용.
  • 일반적인 데이터의 성능을 희생하지 않는 예외 처리 방식.

PHASE 4 | Distributed

Replica Set: High Availability (HA)

  • Failover: Primary 장애 시 12초 내외로 새로운 Primary 선출.
  • Majority Rule: 최소 3대의 노드가 있어야 정상 작동 (홀수 노드 권장).
  • Arbiter: 데이터는 없지만 투표권만 갖는 노드 (비용 절감용).

PHASE 4 | Distributed

Sharding Architecture

  • Mongos: 쿼리 라우터 (클라이언트와 통신).
  • Config Server: 샤드 키 범위 및 위치 정보 저장.
  • Shard Nodes: 실제 데이터를 나누어 저장하는 노드.

PHASE 4 | Distributed

Choosing the Shard Key

  • Cardinality: 값이 다양해야 함 (예: gender는 최악).
  • Frequency: 특정 값에 몰리지 않아야 함.
  • Monotonic Increase: Timestamp 등은 쓰기 시 특정 샤드에만 몰림 (Hashed Sharding 고려).

PHASE 4 | Transaction

Multi-Document ACID Transaction

  • 원격 뱅킹, 주문 시스템 등 무결성이 생명인 곳에서 사용.
  • 단일 문서 원자성은 기본 지원하므로, 여러 문서를 묶을 때만 사용.
  • Snapshot Isolation: 트랜잭션 도중의 변경 사항은 외부에서 보이지 않음.

Anti-Pattern: The Massive Array

DANGER!

문서 내 배열이 계속 커지게 방치하지 마세요. (예: 100만 명의 구독자를 하나의 배열에 저장)

➔ 문서 16MB 제한 초과, 인덱스 성능 급락, 업데이트 지연.

무제한으로 커질 수 있는 데이터는 반드시 컬렉션을 분리(Reference)해야 합니다.

PHASE 6 | Performance

Index ESR Rule Recap

Equality ➔ Sort ➔ Range

// Good Index
{ status: 1, created_at: -1, price: 1 }

// Bad Index (Range before Sort)
{ status: 1, price: 1, created_at: -1 }

순서가 틀리면 인덱스에서 찾은 후 메모리에서 별도의 정렬(In-memory sort)이 일어납니다.
