PART 1 | MongoDB 스키마의 개념

MongoDB에서의 스키마(Schema)란?

  • 스키마(Schema)는 데이터베이스에서 데이터의 구조, 속성, 제약조건 등을 정의하는 설계도입니다.
  • RDBMS(관계형 데이터베이스)는 고정된 스키마를 엄격히 따릅니다.
  • 반면, MongoDB는 스키마리스(Schema-less) 구조를 지원합니다.
    • 유연한 문서 기반(Document-oriented) 구조로, 같은 컬렉션 안의 문서들이 서로 다른 필드를 가질 수 있습니다.
하지만 실제 서비스에서는 데이터 일관성을 위해 Mongoose 같은 ODM을 사용하여 명시적인 스키마를 정의하는 것이 일반적입니다.

PART 1 | MongoDB 스키마의 개념

왜 스키마 디자인이 중요한가?

1
성능 (Performance)
조인(Join)을 최소화하고 쿼리 효율성과 디스크 I/O를 최적화합니다.
2
확장성 (Scalability)
샤딩(Sharding)과 수평 확장, 데이터 파티셔닝에 유리한 구조를 만듭니다.
3
유지보수성
일관된 데이터 구조를 통해 애플리케이션 코드의 안정성을 향상시킵니다.

읽기(Read) vs 쓰기(Write) 비율에 따라 모델링 설계 전략이 완전히 달라집니다!

RDBMS vs MongoDB 모델링

RDBMS (관계형)

  • 테이블 기반
  • 정규화(Normalization) 선호
  • Foreign Key를 통한 조인(JOIN)
  • 스키마가 고정됨

MongoDB (문서형)

  • 문서(Document) 기반
  • 비정규화(Denormalization) 및 임베딩(Embedding) 활용
  • 조인을 피하고 한 번의 읽기로 해결 (Read-optimized)
  • 유연한 스키마(Flexible Schema)

PART 2 | MongoDB Design Pattern

1. Embedding Pattern (중첩 패턴)

  • 관련된 데이터를 하나의 문서에 중첩(Embed)하여 저장하는 방식.
  • 조인(Join) 없이 한 번의 Read로 모든 관련 정보 접근 가능.
  • 장점: 빠른 조회 성능, 데이터 일관성 보장.
  • 단점: 데이터 중복 발생 가능, 16MB 문서 크기 제한 초과 위험.

💡 1:1 관계나 크기가 작은 1:N 관계에 적합 (예: 게시글과 태그)

{ "_id": "post123", "title": "MongoDB 시작하기", "author": { "name": "이경수", "role": "Admin" }, "tags": ["DB", "NoSQL"] }

2. Reference Pattern (참조 패턴)

  • 관련 데이터를 다른 컬렉션에 분리하고, ObjectId를 참조(Reference)하여 연결하는 방식.
  • RDBMS의 Foreign Key 개념과 매우 유사합니다.
  • 장점: 데이터 중복 최소화, 개별 수정/관리가 용이.
  • 단점: $lookup 등 조인이 필요해 다중 쿼리로 인한 성능 저하 가능.

💡 크기가 큰 1:N 관계나 N:M 관계에 적합 (예: 대규모 댓글, 공통 카테고리)

// Category Collection { "_id": "cat_001", "name": "데이터베이스" }
// Post Collection { "title": "MongoDB 시작", "category_id": "cat_001" }

3. Bucket Pattern (버킷 패턴)

  • 고빈도 데이터(이벤트, 시계열 로그 등)를 일정 기준(시간, 유저 등)으로 묶어(Bucket) 하나의 문서에 배열로 저장하는 방식.
  • 문서 수를 획기적으로 줄이고 인덱스 크기를 축소하여 저장 공간 최적화 및 성능 향상.
  • 적용 예: IoT 센서 데이터 수집, 주식 거래 기록, 사용자 활동 히스토리.
Log (12:01)
Log (12:02)
Log (12:03)
➔
{ "sensor_id": 101, "date": "2026-05-27", "measurements": [ {"time": "12:01", "temp": 24.5}, {"time": "12:02", "temp": 24.6}, {"time": "12:03", "temp": 24.5} ] }

4. Computed Pattern (계산 패턴)

  • 계산된 결과 값(Derived value)을 문서에 미리 저장(Precomputed)하는 방식.
  • 복잡한 Aggregation Pipeline 연산 비용을 줄여 Read 성능을 대폭 향상시킵니다.
  • 적용 예: 영화의 총 매출액, 게시글의 총 댓글 수, 상품의 평균 평점 등.
조회수는 높지만, 실시간 100% 정확성보다는 빠른 응답이 중요한 대시보드 및 리포트에 매우 적합합니다.
Revenue = Ticket_Price × Num_Viewers
미리 계산하여 필드에 저장
"total_revenue": 500000

블로그 시스템 요구 명세서

  • 목표: 사용자가 블로그 형식의 게시글을 작성하고, 댓글 및 추천(좋아요)을 주고받는 시스템.
  • 주요 엔티티:
    • 사용자(User): ID, 이름, 이메일, 프로필 이미지, 역할.
    • 게시물(Post): 제목, 본문, 작성자, 첨부 이미지, 상태(공개/비공개).
    • 댓글(Comment): 작성자, 본문, 작성일, 대댓글(부모 ID).
    • 추천(Like): 사용자 1명이 1개의 게시물에 1번만 가능.
    • 카테고리(Category) & 태그(Tag): 게시물 분류 및 검색.

중첩(Nested) vs 참조(Referenced) 실전 판단

중첩 (Embedding) 권장

  • 태그(Tags): 단순한 문자열 배열이므로 Post 내부에 tags: ["MongoDB", "NoSQL"]로 저장.
  • 소규모 댓글: 댓글 수가 적고 항상 게시글과 함께 보여진다면 Post 문서 내 comments 배열로 저장.

참조 (Referencing) 권장

  • 사용자(User): 독립적으로 자주 조회/수정되므로 별도 컬렉션 유지.
  • 카테고리(Category): 재사용되며 사전 정의되므로 참조 사용.
  • 대규모 댓글: 무한 스크롤 등 댓글이 수백 개 달릴 수 있다면 16MB 제한 방지를 위해 Comment 컬렉션으로 분리.

사례 연구 1: 사용자 추천 (Likes) 설계

  • "누가 추천했는지" 중요하지만, 중복 추천을 방지하고 빠르게 추천 수를 세는 것이 주 목적입니다.
  • 관계형 DB처럼 Like 테이블을 따로 만들기보다, Post 내부에 likes: [user_id_1, user_id_2, ...] 배열로 저장하는 것이 훨씬 효율적입니다.
{ "_id": "post_999", "title": "MongoDB 최적화 기법", "likes": [ "user_01", "user_45", "user_77" ], "like_count": 3 // Computed Pattern 적용 }

사례 연구 2: 딥 조인(Deep Join)의 함정

  • User → Post → Comment → Like 까지 연속으로 참조(Reference)로 설계하면 어떻게 될까요?
  • MongoDB는 관계형 DB처럼 완벽하고 자유로운 다중 JOIN 연산에 최적화되어 있지 않습니다.
  • $lookup이 여러 단계 누적되면 조회 속도와 쿼리 가독성이 급격히 저하됩니다.

Anti-Pattern

다단계 참조는 MongoDB 철학(Document-Oriented)에 맞지 않습니다. 적절한 임베딩(비정규화)을 혼합해야 합니다.

읽기 중심 최적화 (Read-centric Optimization)

  • MongoDB는 빠른 읽기를 위해 비정규화(Denormalization) 및 데이터 중복을 적극적으로 활용합니다.
  • 게시물 목록을 보여줄 때 매번 User 컬렉션을 조인하지 않고, 게시물 문서 내부에 작성자의 '이름(username)'이나 '프로필 이미지 URL'을 함께 저장합니다 (마치 캐싱처럼 활용).
주의점: 사용자 닉네임이 변경되면 해당 사용자가 작성한 모든 게시글을 찾아 업데이트해야 하므로 쓰기(Update) 로직은 조금 더 복잡해집니다. 하지만 읽기가 쓰기보다 압도적으로 많기 때문에 유리합니다!
