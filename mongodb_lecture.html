PART 1 | MongoDB 스키마의 개념

MongoDB에서의 스키마(Schema)란?

  • 스키마(Schema)는 데이터베이스에서 데이터의 구조(structure), 속성(attributes), 제약조건(constraints) 등을 정의하는 설계도입니다.
  • 스키마리스(Schema-less): MongoDB는 유연한 문서 기반(Document-oriented) 구조로, 같은 컬렉션 안의 문서들이 서로 다른 필드를 가질 수 있습니다.
  • ODM의 활용: 하지만 데이터 일관성과 검증을 위해 Mongoose 같은 ODM을 사용하여 Node.js 환경에서 명시적 스키마를 정의합니다.
// Mongoose Schema Example const userSchema = new mongoose.Schema({ name: { type: String, required: true }, age: { type: Number, min: 0 }, email: { type: String, unique: true } });
PART 1 | MongoDB 스키마의 개념

왜 스키마 디자인이 중요한가?

1
성능 (Performance)
조인(Join)을 최소화하여 쿼리 효율성을 높이고 디스크 I/O를 최적화합니다.
2
확장성 (Scalability)
수평 확장(Sharding) 및 데이터 파티셔닝에 대비한 구조를 만듭니다.
3
유지보수성
일관된 데이터 구조를 통해 애플리케이션 코드의 안정성을 향상시킵니다.

읽기(Read)와 쓰기(Write)의 비율, 데이터 크기에 따라 설계 전략이 완전히 달라집니다.

PART 1 | MongoDB 스키마의 개념

데이터 모델링 접근: RDBMS vs MongoDB

RDBMS (관계형)

  • 구조: 테이블(Table) 기반
  • 원칙: 데이터 중복을 피하는 정규화(Normalization) 선호
  • 관계: Foreign Key를 통한 JOIN으로 관계 표현

MongoDB (문서형)

  • 구조: 문서(Document) 기반, 중첩 문서와 배열 자유롭게 사용
  • 원칙: 읽기 최적화를 위한 비정규화(Denormalization) 적극 활용
  • 관계: 임베딩(Embedding)과 레퍼런싱(Referencing) 중 선택
PART 1 | MongoDB 스키마의 개념

데이터 모델링: 일대일 (1:1) 관계

  • 사용자(User)와 프로필(Profile), 혹은 환자와 기본 병력 데이터 같은 1:1 관계입니다.
  • Embedding (중첩): 일반적으로 가장 추천됩니다. 하나의 문서 안에 서브 도큐먼트로 저장하여 1번의 쿼리로 모든 정보를 가져옵니다.
  • Referencing (참조): 1:1 관계이더라도 어느 한쪽 데이터가 너무 커서 조회가 느려지거나, 분리해서 권한을 관리해야 할 때만 참조를 사용합니다.
// 1:1 Embedding Example { "_id": "user123", "name": "이경수", "profile": { "age": 30, "address": "Jeonju" } }
PART 1 | MongoDB 스키마의 개념

데이터 모델링: 일대다 (1:N) 관계

  • 1:N 관계는 자식 데이터의 개수(N의 크기)에 따라 모델링 전략이 나뉩니다.
관계 규모 예시 권장 전략 설명
1 to Few (소수) 게시글과 태그(Tags) Embedding 배열 필드에 직접 값을 삽입. 매우 빠르고 관리가 쉬움.
tags: ["MongoDB", "NoSQL"]
1 to Many (다수) 게시글과 수백 개의 댓글 Child Referencing 부모 문서에 자식의 ObjectId 배열을 저장.
comments: [id1, id2, ...]
1 to Squillions (무한대) 시스템 로깅, 센서 스트림 Parent Referencing 자식 문서가 부모의 ObjectId를 가짐 (RDBMS의 FK처럼).
배열 크기 제한(16MB) 방지.

MongoDB Design Pattern

효율적인 데이터 구조를 위한 모델링 패턴

PART 2 | MongoDB Design Pattern

1. Embedding Pattern (중첩 패턴)

  • 관련된 데이터를 하나의 문서에 중첩(Embed)하여 저장하는 방식.
  • 장점:
    • 조인(Join) 불필요, 가장 빠른 조회 성능 (1번의 Read).
    • 단일 문서 단위의 데이터 일관성 보장.
  • 단점:
    • 다른 곳에서도 쓰인다면 데이터 중복 발생 가능.
    • 문서 최대 크기 제한(16MB)을 초과할 위험.
💡 언제 쓸까? 읽기(Read)가 압도적으로 많고, 데이터 크기가 작으며, 자주 변경되지 않는 데이터 (예: 주소, 태그)
{ "_id": "post_01", "title": "Design Patterns", "author": { "name": "이경수", "role": "Admin" }, "tags": ["DB", "NoSQL"] }
PART 2 | MongoDB Design Pattern

2. Reference Pattern (참조 패턴)

  • 관련 데이터를 다른 컬렉션에 분리하고, ObjectId를 참조(Reference)하여 연결하는 방식.
  • 장점:
    • 데이터 중복을 최소화 (정규화).
    • 각 문서를 독립적으로 수정/관리 가능.
  • 단점:
    • $lookup 등 조인이 필요하여 여러 번의 쿼리(fetch) 발생.
    • 복잡한 조회 시 성능 저하 가능.
// Category Collection { "_id": "cat_001", "name": "데이터베이스" }
// Post Collection { "title": "MongoDB 시작", "category_id": "cat_001" }
PART 2 | MongoDB Design Pattern

3. Bucket Pattern (버킷 패턴)

  • 고빈도 데이터(이벤트, 센서, 로그 등)를 일정 기준(시간, 사용자 등)으로 묶어 하나의 문서 배열로 저장하는 방식.
  • 장점: 문서 수 대폭 감소 ➔ 인덱스 크기 축소 ➔ 디스크 I/O 최적화 및 조회 성능 향상.
  • 단점: 문서 크기 16MB 초과 여부 모니터링 필요, 개별 이벤트 단위의 부분 업데이트가 다소 까다로움.
Log (12:01)
Log (12:02)
Log (12:03)
➔
{ "sensor_id": 101, "date": "2026-05-27", "measurements": [ {"time": "12:01", "temp": 24.5}, {"time": "12:02", "temp": 24.6}, {"time": "12:03", "temp": 24.5} ] }
PART 2 | MongoDB Design Pattern

4. Computed Pattern (계산 패턴)

  • 계산된 결과 값(Derived value)을 문서에 미리 저장(Precomputed)하는 방식.
  • 복잡한 Aggregation Pipeline 연산 비용을 줄여 Read 성능을 대폭 향상시킵니다.
  • 적용 예시:
    • 조회수가 엄청난 리포트/대시보드 (총 판매액, 평균 평점 등).
    • 정렬용 기준 필드 (인기순 정렬을 위한 가중치 점수).
데이터 갱신 시 실시간 완벽 동기화가 필요하지 않거나, 주기적인 배치(Batch) 업데이트만으로 충분한 경우 매우 강력합니다.
Revenue = Ticket_Price × Num_Viewers
문서에 필드를 추가하여 직접 저장
"total_revenue": 500000

콘텐츠 플랫폼(블로그) 모델링 실전

PART 3 | MongoDB Practice

블로그 시스템 요구 명세서

  • 목표: 사용자가 블로그 형식의 게시글을 작성하고, 댓글 및 추천(좋아요)을 주고받는 콘텐츠 플랫폼.
  • 주요 엔티티:
    • 사용자(User): 고유 ID, 이름, 이메일, 프로필 이미지, 역할, 가입일시.
    • 게시물(Post): 제목, 본문, 작성자, 썸네일, 첨부 이미지, 상태(공개/비공개/임시).
    • 댓글(Comment): 본문, 작성자, 작성일, 대댓글(선택적 부모 ID).
    • 추천(Like): 사용자 1명이 1개의 게시물에 1번만 추천 제한.
    • 카테고리(Category) & 태그(Tag): 게시물 분류 및 검색을 위한 키워드.
PART 3 | MongoDB Practice

중첩(Nested) vs 참조(Referenced) 실전 판단

중첩 (Embedding) 권장

  • 관련 데이터가 항상 함께 조회되는 경우
  • 크기가 작고 수정될 일이 많지 않은 데이터.
  • 적용 예:
    • 게시글의 첨부 이미지 목록.
    • 게시글 내부의 단순 태그 배열.

참조 (Referencing) 권장

  • 독립적 관리가 필요하거나 확장될 가능성이 높은 경우
  • 적용 예:
    • 사용자(User): 회원 정보는 여러 곳에서 참조되고 수정됨.
    • 카테고리(Category): 사전 정의되어 재사용됨.
    • 댓글(Comment): 무한 스크롤 등 수백 개로 늘어날 위험이 있는 경우.
PART 3 | MongoDB Practice

정규화 vs 비정규화 (Tags 처리)

  • RDBMS에서는 데이터 중복을 막기 위해 교차 테이블(Mapping Table)을 만들어 정규화합니다.
  • MongoDB에서는 읽기 효율을 위해 비정규화(Denormalization) 전략을 적극 사용합니다.
  • 태그는 단순한 키워드이므로 별도 컬렉션으로 분리할 필요 없이 배열로 들고 있는 것이 효율적입니다.

RDBMS (정규화)

Post ➔ PostTag (교차 테이블) ➔ Tag

복잡한 JOIN 연산 필수

MongoDB (비정규화)

Post.tags = ["MongoDB", "NoSQL"]

문서 안에 배열로 단순 저장

PART 3 | MongoDB Practice

사례 연구 1: 소규모 댓글과 추천(Likes) 설계

  • 소규모 댓글: 댓글 수가 적고 항상 게시글과 묶어 보여준다면, Post 안의 comments 배열 객체로 저장하는 것이 효율적입니다. (조인 비용 Zero)
  • 추천(Likes): "누가 추천했는지" 알아야 하고 중복을 방지해야 합니다.
    ➔ likes: [user_id_1, user_id_2] 배열로 저장.
    ➔ RDBMS처럼 Like 매핑 테이블을 만들지 않습니다!
{ "_id": "post_999", "likes": [ "user_01", "user_45" ], "comments": [ { "user": "user_02", "text": "최고의 글!" } ] }
PART 3 | MongoDB Practice

사례 연구 2: 딥 조인(Deep Join)의 함정

  • User → Post → Comment → Like 까지 모든 것을 참조(Reference)로 설계하면 어떻게 될까요?
  • MongoDB는 관계형 DB처럼 완벽하고 자유로운 다중 JOIN을 지원하지 않습니다.
  • $lookup이 여러 단계 누적되면 조회 속도와 쿼리 가독성이 급격히 저하됩니다.

⚠️ Anti-Pattern

다단계 참조는 MongoDB 철학(Document-Oriented)에 맞지 않습니다.

이럴 때는 비정규화(Denormalization)와 임베딩(Embedding)을 혼합하여 참조 단계를 줄여야 합니다.

PART 3 | MongoDB Practice

읽기 중심 최적화 및 확장성 고려

  • 캐싱 형태의 비정규화: 게시물 목록 조회 시, 참조된 User 컬렉션을 매번 조인하지 않고, 작성자의 username이나 profile_url을 게시물에 복사해둡니다.
  • 집계용 필드 동기화: like_count, comment_count를 필드로 미리 만들어(Computed Pattern) 배열 크기를 매번 세지 않도록 설계합니다. 데이터 변경 시 해당 카운트도 $inc 연산자로 동시에 갱신합니다.
// Mongoose 조인(populate) 활용 예제 async function getPostsByCategory(categoryName) { const category = await Category.findOne({ name: categoryName }); if (!category) return []; const posts = await Post.find({ category: category._id }).populate('author_id', 'username'); // author_id를 바탕으로 username만 조인해서 가져옴 return posts; }
